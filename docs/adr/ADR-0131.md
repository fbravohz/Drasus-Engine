### **ADR-0131: Flutter como Framework de Frontend — Rechazo Razonado de Qt, iced, slint y egui**

*   **Decisión:** Se reafirma **Flutter/Impeller** como el framework único de frontend del producto, sobre la integración FFI de [ADR-0029](./ADR-0029.md). Se evalúan y **rechazan** explícitamente las alternativas Qt (C++), `iced`, `slint` y `egui` (Rust). Este ADR existe para cerrar la pregunta "¿deberíamos cambiar de framework de UI?" con su razonamiento documentado, de modo que no se reabra sin contexto nuevo.

*   **Problema que resuelve:** Al delegar todo el desarrollo a agentes de IA, surge la tentación de elegir un framework "más puro" (un solo lenguaje, Rust) o "más robusto" (Qt), bajo la premisa de que la facilidad de programar deja de importar porque ya no programa un humano. Esa premisa está **invertida**: el rendimiento de un agente de IA está acotado por la densidad de su corpus de entrenamiento y la estabilidad de API del framework. Flutter/Dart tiene corpus masivo y API estable; `iced`/`slint`/`egui` tienen corpus escaso y APIs que rompen entre versiones. Delegar a IA es por tanto un argumento **a favor** de Flutter, no en contra.

*   **Por qué Flutter gana en los ejes que importan a un motor cuant retail:**
    *   **Afinidad con desarrollo por IA:** corpus enorme + API estable → menos alucinaciones del agente. Las GUI de Rust de nicho son donde el agente *peor* rinde.
    *   **Charting financiero:** ecosistema maduro (velas, indicadores, profundidad) vía `CustomPainter`/Impeller y librerías existentes; las GUI de Rust tienen charting pobre o de estética de herramienta.
    *   **Tests de UI:** Flutter tiene widget tests, golden tests e integration tests maduros; el testing de UI en `iced`/`slint`/`egui` es inmaduro.
    *   **Optionalidad multiplataforma:** desktop + móvil + web con un solo código, relevante para un futuro compañero móvil o dashboard web del modo SaaS (ver [ADR-0033](./ADR-0033.md)).
    *   **Madurez y licencia:** respaldo de Google, licencia libre (BSD), bajo riesgo de bus factor.

*   **Por qué el beneficio de "un solo lenguaje" vale poco AQUÍ:** la arquitectura define el frontend como **Cáscara Delgada** sin lógica de negocio (FCIS, [ADR-0001](./ADR-0001.md)). La frontera FFI no es solo un costo: **es la costura arquitectónica** que materializa el invariante "Shell delgada / Core puro". Unificar lenguaje paga su mayor valor cuando hay lógica compartida cruzando la frontera; aquí no la hay por diseño. Nota: `slint` tampoco es "un solo lenguaje" — usa su propio DSL declarativo (`.slint`).

*   **Por qué Qt queda descartado por principios propios:** reintroduce **C++**, justo el tipo de residuo no-Rust que el invariante de stack purga ([ADR-0013](./ADR-0013.md), [ADR-0104](./ADR-0104.md)). Su robustez en abstracto no compensa contradecir el invariante; es el peor encaje *para este proyecto*.

*   **Restricciones:**
    *   El frontend NUNCA aloja lógica de negocio — sigue siendo Cáscara Delgada; cambiar de framework no eximiría de necesitar la costura Core↔UI.
    *   Este rechazo es **condicional al perfil de producto actual** (motor cuant retail/solopreneur con UI vendible, charting pulido y modo SaaS). NO es un invariante físico absoluto.

*   **Efecto observable:** la pregunta de migrar de framework de UI no se reabre salvo que cambie el perfil de producto (ver "Cuándo reabrir"). El stack de frontend permanece Flutter/Impeller + `flutter_rust_bridge`.

*   **Cuándo reabrir esta decisión (no es dogma):**
    *   Si Drasus deja de ser un producto retail con UI vendible y se vuelve una terminal/herramienta interna de desarrollo sin charting pulido ni móvil/web → `egui` sería candidato fuerte (iteración rápida, Rust puro, ideal para HUDs de depuración).
    *   Si se decide formalmente que NUNCA habrá móvil ni web y se acepta la inmadurez de charting y tests de UI a cambio de un toolchain único → `iced` sería candidato.
    *   Ninguno de esos escenarios encaja con el perfil de producto vigente.

*   **Costo del cambio rechazado:** migrar no sería "cambiar el frontend" sino reescribir el **corpus de diseño entero** — los slices de UI/FFI de ~138 features, los skills `bridge-engineer` y `flutter-engineer`, los ADR que fijan Flutter/Impeller/`flutter_rust_bridge`, el invariante de `CLAUDE.md §1`, el SAD y el ROADMAP. Esa migración la ejecutarían agentes sobre el framework donde peor rinden: doble penalización.

*   **Trazabilidad:** Extiende [ADR-0029](./ADR-0029.md) (Patrón Todo en Uno Rust + Flutter FFI).
