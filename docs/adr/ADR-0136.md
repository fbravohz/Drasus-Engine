### **ADR-0136: Arquitectura Visual Unificada — Dashboard + Canvas [Forge/Reactor — TBD] (supersede ADR-0028)**

*   **Decisión:**
    La interfaz visual de Drasus Engine se reorganiza en dos superficies distintas: (1) **Dashboard** — centro de monitoreo y navegación estática, y (2) **Canvas unificado [Forge/Reactor — nombre TBD]** — lienzo nodal con zoom de dos estados. Esta arquitectura supersede ADR-0028 (ZUI Fractal de 3 niveles nominados). Los términos "MACRO", "MESO" y "MICRO" como *nombres de nivel de navegación* quedan descontinuados. El paradigma del Zoomable UI se conserva, pero como zoom continuo dentro del canvas, no como tres pantallas o vistas discretas con nombres fijos.

*   **Contexto y problema que resuelve:**
    ADR-0028 definía tres niveles nominados (Fleet Command MACRO → Orchestrator MESO → Strategy Inspector MICRO) que colisionaban con la realidad: el DAG de nodos no pertenece a "un nivel" — aparece en todos (flota de estrategias, pipeline de módulos, lógica interna de una estrategia). Los nombres forzaban una taxonomía de pantallas que no reflejaba cómo el usuario piensa en el sistema. La nueva arquitectura reemplaza los nombres de nivel por una jerarquía de entidades expresada mediante anidación de card-nodes, accesible desde un canvas infinito con zoom in-place.

*   **Superficie 1 — Dashboard (centro de control)**
    - Pantalla de inicio. Centro de monitoreo. **No es un nivel de zoom ni participa en el canvas.**
    - Contiene widgets de features: métricas, KPIs, estado del sistema — todos **read-only**. La edición siempre ocurre en el canvas.
    - **Personalizable por el usuario:** arrastra widgets de features y los organiza en bento grid (cuadrícula flexible). Un widget es la "vista de monitoreo" de una feature específica.
    - Todo elemento clickeable navega al canvas en el contexto correspondiente (portafolios, estrategias, módulos, pipelines). El Dashboard nunca hace zoom propio.
    - Algunos elementos del Dashboard son estáticos (solo números); otros son navegables (módulos, features, estrategias, portafolios, clusters).

*   **Superficie 2 — Canvas unificado [Forge/Reactor — TBD]**
    - Lienzo infinito único. Paradigma: card-nodes rectangulares conectados mediante bezier S-curves con puertos tipados.
    - **Unificado:** el mismo canvas contiene nodos de entidad (Cluster, Portfolio, Strategy) y nodos de proceso (Módulo, Feature). No hay dos canvas separados.
    - **Focus mode (capa de visibilidad):** al abrir el canvas desde el Dashboard en contexto "Portafolios", el canvas filtra para mostrar nodos de entidad. En contexto "Módulos/Pipeline", muestra nodos de proceso. El usuario puede desactivar el filtro para ver el grafo completo. El filtro no es una vista diferente — es visibilidad sobre el mismo canvas.
    - Fondo: `deepSpace #080A18` + dot-grid puntos 1.5px `borderPanel #1B2440` @ 20px (estilo N8N/Figma). Sin líneas de cuadrícula.

*   **Modelo de zoom — dos estados, zoom continuo**
    - **Vista relacional** (alejado): card-nodes en miniatura, conexiones visibles, topología del grafo. Aquí el usuario conecta, desconecta, reordena nodos. Los nodos muestran información mínima (nombre, estado semántico, chip de régimen).
    - **Vista interior** (acercado / doble clic en un nodo): el nodo se expande in-place. Los nodos del contexto padre se desplazan al borde del viewport ("ventana de submarino") — dan contexto de dónde estás sin salir del canvas. El usuario edita el contenido interno del nodo.
    - **Transición:** animación de zoom in-place (`InteractiveViewer` de Flutter). El canvas no cambia de pantalla; la cámara se mueve.
    - El zoom es **continuo** dentro de cada estado, como Figma o Google Maps. No hay límites discretos artificiales.
    - **No hay un tercer estado:** la jerarquía Cluster→Portfolio→Strategy→Logic Blocks se navega como sucesivos "entrar en nodo" en el mismo canvas. El breadcrumb da el contexto de profundidad.

*   **Navegación — Breadcrumb flotante**
    - Barra de vidrio Apple flotante en la parte superior del canvas: `Cluster A › Portfolio B › Strategy 3`.
    - Clic en cualquier segmento → zoom out hasta ese nivel, con animación in-place.
    - Obligatorio cuando hay ≥1 nivel de anidación activo.

*   **Jerarquía de entidades (nodos de entidad)**
    ```
    Cluster (nivel 0)
      → Portfolio × N
          → Strategy × N
              → Logic Blocks × N (DAG de señales/indicadores/ejecución)
    ```
    La jerarquía se expresa mediante **anidación in-place**: hacer zoom en un Cluster revela sus Portfolios; zoom en un Portfolio revela sus Strategies. Los nodos de un nivel **no coexisten** con los del siguiente en la misma vista — el zoom los separa naturalmente.

*   **Jerarquía de proceso (nodos de módulo/feature)**
    Los pipelines de proceso (Ingest→Withdraw, crypto pipeline, forex pipeline, etc.) viven en el mismo canvas como secuencias de nodos de módulo y/o feature conectados por tipos de dato compatibles (ADR-0137).
    - **Módulo (compound node):** nodo que contiene features con cableado preset. Usable como caja negra o expandible para ver sus features internas. Los 8 módulos del pipeline son el **preset default** — la ruta recomendada para principiantes.
    - **Feature (atomic node):** unidad mínima de proceso. Tiene puertos de entrada y salida tipados (ADR-0137). Al hacer clic abre un **inspector panel lateral** (derecha, estilo N8N) con la UI específica de la feature. No abre pantalla full.
    - **Custom module:** composición de features guardada por el usuario como nodo compuesto reutilizable. El sistema valida compatibilidad de tipos antes de guardar.
    - Usuario experto: puede ignorar módulos completos y cablear features directamente (ver ADR-0137).

*   **Comportamiento de apertura por tipo de nodo**

    | Tipo de nodo | Tiene sub-nodos | Cómo se abre |
    |---|---|---|
    | Cluster | Sí (portfolios) | Zoom in-place en el canvas |
    | Portfolio | Sí (strategies) | Zoom in-place en el canvas |
    | Strategy | Sí (logic blocks) | Zoom in-place → DAG de lógica interna |
    | Módulo | Sí (features) | Zoom in-place → vista de módulo |
    | Feature | No (leaf node) | Inspector panel lateral (right-side) |
    | Logic block | No (leaf node) | Inspector panel lateral |

    **Vista de módulo (mixer view + graph view):** al expandir un módulo, la vista interna por defecto es un "mixer" estilo DAW — features como canales con estado activo/inactivo, flujo de datos en tiempo real, toggle de activación. Toggle disponible: `graph view` (features como sub-nodos DAG con cableado visible) ↔ `mixer view` (canales estilo DAW). Ambas vistas son equivalentes; el usuario elige la que prefiere.

*   **Conexiones — bezier S-curve con tipos semánticos**
    - Curvas bezier cúbicas (S-curve estilo N8N): puntos de control separados horizontalmente desde cada puerto.
    - Cada línea tiene un **tipo de dato** (definido en ADR-0137). Color de la línea = color del tipo.
    - Conectar output de tipo T₁ a input de tipo T₂ incompatible → conexión marcada `criticalCrimson` parpadeante + tooltip "tipos incompatibles".
    - Núcleo: `strokeWidth 2` + halo de glow `blur 4`. Punta de flecha 8px en el nodo destino.
    - Hover sobre línea: `strokeWidth 3` + glow intensificado + tooltip con tipo de dato y nombre de la conexión.

*   **Anatomía del card-node** (especificación visual — ver también `docs/DESIGN.md §Components §Grafo/DAG`)
    - Cuerpo: `cardInner #11182E`, radio 10px, hairline `borderPanel #1B2440`.
    - Header (32–36px): borde izquierdo 3px en color de estado + icono + nombre en `displayGrotesque 13px 500` + chip de tipo.
    - Body: key-values (label: `textLabel`, valor: `dataMono 12px`). Máx 4–5 pares antes de colapsar.
    - Puertos: círculos 10px en borde lateral (cardInner + anillo 1.5px del color del tipo). Izq = input, Der = output. Múltiples puertos permitidos.
    - Estado semántico: borde 1px reposo → borde 2px activo/seleccionado + `glow(color)`. Inválido: borde `criticalCrimson` parpadeante + `glowStrong`. Procesando: `scanRing(estadoColor)` en puerto de salida. Recibe dato: `sonarPulse(estadoColor)` en puerto de entrada.

*   **Reglas irrompibles (FIJO)**
    - Un feature node siempre abre inspector panel lateral. Nunca pantalla full separada.
    - El Dashboard es read-only. Cero edición de lógica en él.
    - Breadcrumbs son obligatorios con ≥1 nivel de anidación activo.
    - Conexiones sin tipo declarado (ADR-0137) son inválidas.
    - El canvas es único (no dos canvas). El focus mode es solo visibilidad.

*   **Supersesión y descontinuación**
    - **ADR-0028 (ZUI Fractal de 3 niveles):** supersedido. Los nombres "Fleet Command (MACRO)", "Orchestrator (MESO)" y "Strategy Inspector (MICRO)" se descontinúan como nombres de nivel de navegación. La funcionalidad que representaban se expresa mediante la anidación de card-nodes en el canvas.
    - Referencias en documentos a "nivel MACRO/MESO/MICRO" deben actualizarse: usar "Vista relacional / Vista interior" o el nombre de la entidad (Cluster, Portfolio, Strategy) según el contexto.
    - Los alias informales prohibidos por ADR-0038 quedan doblemente prohibidos.
