### **ADR-0152: Banco de Pruebas como Gate Permanente de Definición de Terminado — Realización Canónica de la Superficie de Verificación Funcional Genérica**

*   **Decisión:**
    A partir de ahora, **ninguna Feature con superficie propia se considera Terminada (cierre de Etapa 5/QA) si no tiene su conexión individual al Banco de Pruebas** — el harness SVF genérico que vive en `ui/lib/tabs/verification_bank/` y se formaliza como feature en [`verification-bench`](../features/verification-bench.md). El Banco de Pruebas deja de ser "una entrega más" dentro del alcance de cada Story y pasa a ser la **realización canónica y permanente** de la Superficie de Verificación Funcional (SVF) que ADR-0117 ya exige: el punto único, gráfico, donde el propietario (perfil frontend, sin canal práctico de terminal) confirma con el ratón que el backend real de una feature funciona.

*   **Contexto y problema:**
    ADR-0117 ya obliga a que cada feature con superficie propia entregue su SVF junto al backend, bajo un Techo Fijo. Pero el gate de cierre (Etapa 5, Tech-Lead) verificaba cada SVF a mano, feature por feature, sin exigir que todas pasaran por el **mismo** mecanismo. El resultado observado: los **14 cimientos del substrato de monetización** (ADR-0144/0145/0146/0147/0148/0149) tienen backend con QA APTO, verificado por mutación — pero **ninguno** tiene conexión al Banco de Verificación de la UI. Solo son verificables por CLI (Canal #2, ADR-0142). Esta brecha está rastreada como `DEBT-005` desde el 2026-07-04, con el disparador "al cerrar los backends del substrato" — disparador que ya se cumplió (substrato 14/14 cerrado, auditoría retroactiva ejecutada 2026-07-10). Sin un gate explícito y permanente en el proceso de cierre, el patrón se repite indefinidamente: cada tanda nueva de features "backend-first" acumula deuda de UI de verificación porque nada obliga a pagarla antes de seguir avanzando.

*   **Modelo canónico del Banco de Pruebas (dictado por el propietario — detalla el "Banco de Verificación" que ADR-0117 ya nombra):**
    - **Selector de feature:** lista lateral, una entrada por feature verificable, punto de extensión único (el registro central).
    - **Panel de detalle en tres zonas:** input JSON (o estructura equivalente) precargado y editable a la **izquierda**; botón **Enviar** al **centro**; respuesta real del backend por FFI, read-only, a la **derecha**. Es la gemela gráfica de `drasus verify`.
    - **Requisito nuevo — validez de forma visible (FIJO):** la respuesta muestra siempre, de forma visual e inequívoca, si el input estaba bien o mal formado para el puente FFI (Válido / Inválido + razón concreta) — **no solo** el resultado de negocio (`ok`/salida/error). El harness valida la estructura del input contra lo que la feature espera **antes** de disparar la llamada real, y señala el resultado de esa validación con claridad.
    - **Backend real, siempre local:** corre contra el backend real de la propia máquina del usuario por FFI — nunca contra el servidor central/Cabina de Mando (Plano #3, ADR-0143). Esa conexión de red es, en el mejor de los casos, un cable final diferido, fuera del alcance de este ADR.

*   **Reglas:**
    - **Gate de Definición de Terminado (FIJO):** ninguna Feature con superficie propia declarada cierra Etapa 5 (QA) sin su conexión individual al Banco de Pruebas. Esta regla especializa, para el caso concreto del harness genérico, el "Gate de Integración Anti-Deuda" que ADR-0117 ya establece — no lo reemplaza, lo hace exigible con un mecanismo único y verificable en vez de una revisión caso por caso.
    - **Un solo punto de extensión (FIJO):** conectar una feature nueva es añadir una entrada al registro central del Banco (`kVerificationRegistry` o su sucesor) — nunca tocar el layout del tab maestro-detalle, nunca duplicar el harness a medida por feature.
    - **Validación de forma antes que resultado de negocio (FIJO, requisito nuevo):** un input inválido nunca dispara la llamada real al backend. La Zona Derecha distingue siempre "input mal formado" de "input válido pero la operación falló" — son dos estados distintos, con dos tratamientos visuales distintos.
    - **Datos reales, nunca mock (hereda ADR-0117):** la Zona Derecha siempre muestra la respuesta real vía FFI de la función de verificación de la feature — la misma que invoca el CLI (`drasus verify`, Canal #2, ADR-0142). El Banco de Pruebas despacha a esa función; nunca reimplementa su lógica.
    - **Independencia de canales (hereda ADR-0142):** el Banco de Pruebas (Canal #1 gráfico) y el CLI (Canal #2) invocan la misma `public_interface`. Un fallo en uno no bloquea al otro.
    - **Sin puerto de Alpha en el Canvas (hereda ADR-0137):** el Banco de Pruebas es plomería crosscutting — no produce un tipo de dominio propio, no tiene nodo en el Canvas [Forge/Reactor]. Vive únicamente dentro del Panel Operativo Fundacional, mismo criterio de residencia (`crates/shared`) que `clock`/`audit-log`/`telemetry` y los 14 cimientos del substrato.

*   **Implementación:**
    La feature [`verification-bench`](../features/verification-bench.md) formaliza el chasis ya nacido en EPIC-0/SPIKE-006 (tab maestro-detalle + registro extensible, hoy con una sola feature conectada a mano: `sovereign-data-fetcher`) y le añade lo que falta: un despachador genérico FFI (identificador de feature + JSON de entrada → estado del input Válido/Inválido + resultado general + salida real + error) y la validación visual de forma. El **sprint que abre este ADR** conecta retroactivamente las verificaciones que hoy solo existen por CLI — los **14 cimientos del substrato de monetización** — y cualquier otra feature con backend terminado que aún no tenga su conexión individual. Esta conexión retroactiva paga `DEBT-005`.

*   **Costo:**
    El despachador genérico se construye una sola vez (costo ya estimado y acotado en `verification-bench.md`, TTR-001/TTR-002). A partir de ahí, cada feature nueva con superficie propia añade, sobre lo que ADR-0117 ya exige, solo el trabajo de registrar su entrada y declarar su input de ejemplo — "casi gratis" por feature, tal como lo definió el propietario. El costo de gobernanza es el gate mismo: Etapa 5 (Tech-Lead) añade una verificación de cierre más a su checklist, sin trabajo de implementación nuevo por sí sola.

*   **Trazabilidad:** ADR-0117 (enmendado por este ADR — el Banco de Pruebas es la realización canónica de la SVF genérica), ADR-0136 (Canvas, contexto de superficie), ADR-0137 (puertos tipados, tipo `textLabel`, criterio de residencia crosscutting), ADR-0142 (Canal #2 CLI, misma `public_interface`), ADR-0143/ADR-0144/ADR-0145/ADR-0146/ADR-0147/ADR-0148/ADR-0149 (los 14 cimientos que este sprint conecta retroactivamente), `DEBT-005`, [`verification-bench`](../features/verification-bench.md).
