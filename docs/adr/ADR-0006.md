> 🔶 **Enmendado por ADR-0141** — añade las reglas de contenido de las migraciones (tipos canónicos, STRICT mode, PKs UUIDv7, convenciones). ADR-0006 sigue rigiendo la cadena y el mecanismo de aplicación.

### **ADR-0006: Migraciones Centralizadas con SQLx Migrator**

*   **Decisión:** Todas las migraciones de base de datos (cambios en estructura de tablas) se administran desde un único lugar centralizado mediante **SQLx Migrator**, compilando los archivos de migración directamente en el ejecutable final.
*   **Objetivo:** Garantizar que todos los ambientes locales y remotos tengan exactamente el mismo esquema de base de datos sin requerir intérpretes externos (como Python/Alembic) ni dependencias de runtime pesadas.
*   **Implementación:**
    *   Una carpeta `/migrations` en la raíz del proyecto contiene archivos `.sql` secuenciales (ej. `0001_init.sql`, `0002_jobs.sql`).
    *   La macro `sqlx::migrate!("./migrations")` embebe físicamente los scripts SQL dentro del binario de Rust durante la compilación.
    *   Durante el ciclo de inicialización del Core (antes de habilitar la comunicación vía FFI con Flutter), el orquestador ejecuta la migración programáticamente en la base de datos SQLite con `sqlx::migrate!().run(&pool).await`.
    *   Una única cadena de cambios lineal (sin bifurcaciones) controlada por timestamps/secuencias.
*   **Reglas Críticas:**
    *   Si un cambio ya se aplicó, su archivo de migración no se edita en caliente; se crea un nuevo archivo de migración secuencial para aplicar la modificación.
    *   Todos los esquemas de tablas definidos por los módulos se unifican bajo este directorio de migraciones centralizado.
    *   Cada migración debe ser determinista e idempotente.
*   **Ventaja:** Cero dependencias de Python (Alembic) en runtime; validación y ejecución de migraciones instantáneas en el arranque del cliente local; esquema inmutable dentro del binario de distribución.
*   **Costo:** Requiere recompilar el binario de Rust si se añade una nueva migración (comportamiento estándar en desarrollo nativo).
*   **Resultado:** Esquema de base de datos local SQLite siempre sincronizado automáticamente al iniciar la aplicación, sin fricción de instalación para el usuario final.

#### Enmienda 2026-06-28 — Política Greenfield/Brownfield y Modelo de Producción Desktop

**Modelo de producción:** Drasus Engine es un monolito de escritorio. "Producción" es la instancia instalada en la máquina de cada usuario individual — no un servidor central ni un servicio compartido. No existe un único esquema de base de datos compartido que deba migrar en caliente.

**Fase GREENFIELD (pre-release):** mientras ningún usuario final ejecute una build distribuida, el baseline de migraciones es **editable in-situ**. En esta fase:
- Recrear una tabla añadiendo `STRICT`, cambiando tipos o corrigiendo columnas es legítimo y es la forma correcta de corregir el baseline.
- La regla "si un cambio ya se aplicó, no se edita en caliente" aplica solo en BROWNFIELD (ver abajo).
- El criterio para saber si el proyecto está en GREENFIELD: ningún artefacto ejecutable ha sido distribuido a un usuario final externo.

**Congelamiento (transición a BROWNFIELD):** se dispara con el **primer release distribuido** — el primer binario firmado entregado a un usuario externo. A partir de ese momento:
- El baseline es inmutable. Las reglas críticas originales de este ADR entran en vigor sin excepción.
- Toda corrección o extensión del esquema se realiza con **migraciones forward-only**: archivos nuevos y secuenciales, nunca edición de los existentes.
- Las migraciones deben ser **robustas a saltos de versión**: un usuario puede actualizar de v1.0 directamente a v3.0 sin haber ejecutado las versiones intermedias. SQLx Migrator garantiza esto por diseño (cadena lineal, ejecuta solo lo pendiente).
- `ON DELETE CASCADE` en migraciones de corrección: PROHIBIDO. La inmutabilidad del event-store es un invariante físico anterior a cualquier política de migración.
