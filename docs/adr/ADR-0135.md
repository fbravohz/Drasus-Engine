### **ADR-0135: Sección `## Cáscara Visual` Obligatoria en Features con Superficie de Usuario — Skill UI-Designer como Etapa 0.5 del Tech Lead**

*   **Decisión:**
    Toda feature que declare "Superficie propia" en su "Contrato de Integración UI" (ADR-0117) **debe** tener una sección `## Cáscara Visual (Thin Shell)` que especifique el diseño visual con el vocabulario canónico de `docs/DESIGN.md` y los IDs del catálogo `docs/DESIGN.md §"Catálogo de Componentes"`. Esta sección es producida por el skill `ui-designer` (`.claude/skills/ui-designer/SKILL.md`) en la **Etapa 0.5** del pipeline del Tech Lead — después de seleccionar el TTR (Etapa 0) y antes de la implementación Rust (Etapa 2) y antes de que el Flutter Engineer reciba la feature (Etapa 4).

*   **Objetivo:**
    Cerrar la brecha entre la especificación funcional (responsabilidad del Architect/Quant) y la implementación visual (responsabilidad del Flutter Engineer). Sin esta sección, el Flutter Engineer recibe "aparece un chip verde" cuando necesita saber que es `text: #54E8D0, fondo: #08251F, borde: 1px #1E5E4F, radio: 8px, glow(optimaCyan)`. La ambigüedad produce inconsistencia visual entre features, interpretaciones incompatibles con el sistema de diseño y retrabajo de UI en EPIC-8.

*   **Reglas:**
    - **Obligatoria en features con superficie (FIJO):** el Tech Lead no despacha al Flutter Engineer (Etapa 4) si la feature tiene "Superficie propia" y la sección no existe o está desactualizada. Sin sección, la Etapa 4 está bloqueada.
    - **Solo tokens canónicos (FIJO):** la sección usa exclusivamente tokens de `docs/DESIGN.md` y IDs de componentes de `docs/DESIGN.md §"Catálogo de Componentes"`. Prohibido usar hex sueltos inventados o componentes no catalogados.
    - **Color semántico, nunca decorativo (FIJO):** cada token de vitalidad debe tener un estado de negocio asociado (óptimo / transición / alerta / crítico). Prohibido "cian porque se ve bonito".
    - **Producida por ui-designer, no por el Architect (FIJO):** el Architect define la arquitectura de UI (ZUI, tokens, ADR-0106/0028); el ui-designer aplica esa arquitectura a cada feature específica. Son roles distintos — el Architect no reemplaza al ui-designer y viceversa.
    - **Features de plomería (CONFIG):** las features que declaran "Ventana de Verificación" (sin superficie propia) no requieren la sección completa — solo una nota breve indicando el observable que debe aparecer en la feature consumidora.
    - **Omisión permitida (CONFIG):** si la sección ya existe, está actualizada (post-2026-06-22, fecha de finalización de `DESIGN.md`) y no hubo cambios en el sistema de diseño desde entonces, el Tech Lead puede omitir la Etapa 0.5 para esa feature.

*   **Contenido mínimo de la sección `## Cáscara Visual`:**
    1. **Contexto de superficie (ADR-0136):** Dashboard widget / Canvas Vista Relacional / Canvas Vista Interior / Inspector Panel asignado, con descripción en una frase.
    2. **Superficie y densidad:** panel-solid vs panel-glass, densidad (densa / cómoda).
    3. **Componentes:** tabla con IDs del catálogo, rol en la feature, tokens clave y estados semánticos.
    4. **Estados semánticos:** mapeo de estado de negocio → token de vitalidad → tratamiento visual completo (texto, fondo, borde, radio, glow).
    5. **Layout:** grid, agrupaciones, medidas clave.
    6. **Animaciones aplicables:** selección de las reglas de Motion Philosophy de `DESIGN.md` que aplican a esta feature.
    7. **Notas de implementación:** instrucciones específicas para el Flutter Engineer que no encajan en las tablas.

*   **Posición en el archivo de feature:**
    La sección `## Cáscara Visual (Thin Shell)` se ubica **después de `## Tareas (TTRs)` y antes de `## Gobernanza y Estándares`**.

*   **Violaciones que el ui-designer corrige como parte de su trabajo (pre-diseño):**
    - Menciones de WebGL / WebView / DOM / HTML / CSS / SVG → `CustomPainter`/Impeller + datos downsampled de Rust (ADR-0097/ADR-0116).
    - Cálculos analíticos en el frontend → "Core calcula, Flutter visualiza" (ADR-0106).
    - Alias informales de contexto de UI → nomenclatura institucional de ADR-0136/ADR-0038.
    - Referencias a tecnologías rechazadas en la UI → eliminadas (CLAUDE.md §1, ADR-0131).

*   **Implementación:**
    - El skill `ui-designer` (`.claude/skills/ui-designer/SKILL.md`) implementa este ADR con su pipeline de 6 pasos.
    - La plantilla `docs/templates/FEATURE.md` incluye la sección `## Cáscara Visual` con su estructura canónica.
    - El Tech Lead añade la verificación de la sección en su Gate de Coherencia Pre-Despacho (§"Órdenes de Trabajo", Etapa 0).

*   **Costo:**
    Una invocación del ui-designer por feature, una sola vez, antes de la primera implementación de su Cáscara Delgada. El costo marginal (tiempo de una invocación de Sonnet) es insignificante comparado con el retrabajo de UI por especificación visual ambigua y la inconsistencia entre features en EPIC-8.

*   **Trazabilidad:** ADR-0028 (supersedido por ADR-0136), ADR-0038, ADR-0097, ADR-0106, ADR-0116, ADR-0117, ADR-0131, ADR-0136, `docs/DESIGN.md`, `docs/DESIGN.md §"Catálogo de Componentes"`, `docs/templates/FEATURE.md`, `.claude/skills/ui-designer/SKILL.md`, `.claude/skills/tech-lead/SKILL.md`.
