### **ADR-0123: Cabina Dual — Acceso Agéntico vía MCP con Permisos Graduados por Riesgo de Pipeline**

*   **Decisión:**
    Se expone el Core de Rust a un protocolo de control agéntico (MCP) como un **Shell-cliente más**, al mismo nivel que el cliente Flutter (FFI) y el cliente Headless remoto (gRPC, ADR-0033). Cualquier operación que hoy un humano dispara desde la interfaz Flutter sobre la `public_interface` de un módulo queda igualmente disponible para un agente LLM externo (ej. Claude) conectado vía MCP. El usuario decide, en cada momento, qué mano opera — la suya vía interfaz, o la del agente vía MCP — sin que una sustituya arquitectónicamente a la otra (Cabina Dual). Los permisos del canal MCP se gradúan por defecto según el riesgo del pipeline invocado:
    - **Abierto por defecto:** `ingest`, `generate`, `validate`, `incubate`, `feedback` — el agente puede leer, escribir y ejecutar el flujo completo de descubrimiento y simulación sin gate humano por cada paso.
    - **Condicionado por dato:** `manage` — el módulo no separa una porción "de producción" en su lógica (gestiona pesos y reglas de forma idéntica para cualquier portafolio); la frontera real ya existe como dato: el campo `institutional_tag` (Live / Demo, ADR-0020) del portafolio objetivo de la llamada. Si el objetivo es Demo/Paper, la llamada pasa libre; si es Live, se bloquea salvo que el interruptor de producción esté activo.
    - **Bloqueado por defecto:** `execute`, `withdraw` — ambos operan exclusivamente con capital y broker reales (fills reales, cierre de posiciones reales); ninguna conexión MCP nace con permiso sobre ellos.
    El bloqueo de producción se levanta exclusivamente mediante un interruptor explícito y reversible del propietario del despliegue local. En modo SaaS (ADR-0033, SaaSCloudEngine) ese mismo interruptor solo queda disponible después de que el usuario acepta de forma expresa los términos de riesgo de delegar producción a un agente LLM.

*   **Objetivo:**
    Resolver la disyuntiva entre "SaaS de asientos" (el usuario hace todo el trabajo manual dentro de la interfaz) y "software por resultados" (las tareas tediosas de descubrimiento — lanzar barridos, iterar AST, revisar reportes — las puede hacer un agente, o el usuario, y el usuario decide cuál). Esto no diluye la Soberanía sobre el capital real: el agente nunca nace con autoridad sobre producción; solo la ejerce si el dueño se la delega explícitamente, de la misma forma en que delegaría una llave física.

*   **Reglas:**
    - **Cliente, no lógica nueva (FIJO):** el canal MCP consume la misma `public_interface` que ya usan Flutter y el gRPC Headless. Prohibido crear un camino de validación paralelo o lógica de negocio exclusiva para el agente — violaría FCIS (la lógica vive en la Feature, no en el Shell que la invoca) y la Soberanía de Datos (ADR cross-module).
    - **Bloqueo total de `execute`/`withdraw` por defecto (FIJO):** ninguna conexión MCP nace con permiso sobre fills reales, envío de órdenes al broker real o cierre de posiciones reales.
    - **Permiso condicionado por `institutional_tag` en `manage` (FIJO):** la discriminación Live/Demo se resuelve por el dato del portafolio objetivo, nunca por una distinción estructural inventada dentro del módulo.
    - **Interruptor de producción (CONFIG):** activación explícita, oculta hasta que el propietario la activa, reversible en cualquier momento. En SaaS, requiere aceptación expresa de términos de riesgo antes de existir para esa cuenta.
    - **Procedencia auditable (FIJO):** toda acción ejecutada vía MCP queda registrada con el mismo rigor que una acción humana (Perfil Ops/Auditoría, ADR-0020) y marcada como originada por agente, nunca indistinguible de una acción humana.
    - **Excepción acotada, no derogación (FIJO):** las restricciones puntuales que hoy prohíben que un LLM decida o fije parámetros de producción por sí solo (ADR-0115; restricción de `plateau-copilot.md`) siguen siendo el comportamiento de fábrica. Este ADR no las deroga — abre la única vía formal por la que el propietario puede delegar esa autoridad, y solo mientras el interruptor está activo.

*   **Implementación:**
    El propietario conecta un agente LLM a Drasus Engine vía MCP igual que conectaría cualquier otra herramienta MCP. El agente ve expuestas las fronteras públicas de `ingest`/`generate`/`validate`/`incubate`/`feedback` como herramientas invocables y opera el flujo de descubrimiento sin pedir permiso por paso. Si intenta tocar `manage` sobre un portafolio Live, o cualquier frontera de `execute`/`withdraw`, sin que el interruptor esté activo, el sistema rechaza la llamada con el mismo aparato de RBAC que ya protege el Gateway SaaS. El propietario puede cerrar el canal MCP en cualquier momento y seguir operando 100% manual desde Flutter — es modo dual, nunca modo exclusivo.

*   **Costo:**
    Requiere construir, para el canal MCP, el mismo aparato de auditoría/RBAC que ya protege gRPC (Gateway SaaS), más el interruptor de activación de producción y su flujo de aceptación de términos. No agrega lógica de negocio nueva — reutiliza el FCIS existente de cada módulo.
