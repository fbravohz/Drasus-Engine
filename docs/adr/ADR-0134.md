### **ADR-0134: Matriz de Plataformas de Despliegue (Desktop Nativo + Mobile/Web Cliente Delgado) y Detección de Muerte del Padre Portátil**

*   **Decisión:** Fijar canónicamente en qué sistemas operativos corre cada mitad del sistema y por qué canal se conectan. Dos planos:
    1.  **Backend Rust (Core, los 8 módulos):** corre **nativo** en los tres sistemas de escritorio de producción — **Windows, Linux y macOS** — y en **servidor Linux** (VPS). **NUNCA** corre en móvil (iOS/Android) ni en navegador. Es decir: el binario Core debe compilar y funcionar idénticamente en Windows, Linux y macOS; ninguno es plataforma "secundaria".
    2.  **Frontend Flutter (cáscara delgada):** corre en los tres escritorios (junto al Core, vía FFI) y, además, como **cliente delgado puro** en **móvil** (iOS/Android) y —diseño preparado, implementación futura incierta— en **Web**. El cliente delgado no aloja lógica: se conecta a un Core Rust remoto o local por **gRPC**.

*   **¿Por qué lo decidimos?** Hasta ahora ningún documento declaraba la matriz de sistemas operativos. ADR-0033 define los *modos* de despliegue (cómo se acoplan UI y Core), pero no qué SO son targets de producción ni dónde vive el cliente móvil. Esa omisión causó un bug real: un worker usaba una llamada al kernel exclusiva de Linux dentro de un bloque marcado como "cualquier Unix", que incluye macOS — y rompía la compilación en Mac. Declarar la matriz explícitamente convierte "soportar macOS" en un invariante verificable, no en un supuesto.

*   **Matriz canónica (qué corre dónde):**

    | Plataforma | Backend Rust (Core) | Frontend Flutter | Acoplamiento |
    |---|---|---|---|
    | Windows (escritorio) | Sí, nativo | Sí | FFI mismo proceso (modo `LocalPowerUser`, ADR-0033) |
    | Linux (escritorio) | Sí, nativo | Sí | FFI mismo proceso (`LocalPowerUser`) |
    | macOS (escritorio) | Sí, nativo | Sí | FFI mismo proceso (`LocalPowerUser`) |
    | Servidor Linux (VPS) | Sí, daemon headless | No (o cliente remoto) | gRPC (modo `SaaSCloudEngine`, ADR-0033) |
    | iOS / Android (móvil) | No | Sí, cliente delgado | gRPC a Core remoto/local |
    | Web (navegador) | No | Futuro incierto, cliente delgado | gRPC a Core remoto |

*   **Topología del cliente móvil:** la app Flutter móvil es cliente puro y se conecta a un Core Rust que vive en uno de tres sitios, los tres válidos: (a) un **VPS** del usuario, (b) una **red de VPS distribuida** (la topología de planos Control/Ejecución de ADR-0119), o (c) el **escritorio del propio usuario por red local**. El móvil nunca ejecuta los módulos; solo manda comandos y pinta telemetría. Es la extensión natural del modo `SaaSCloudEngine` de ADR-0033: el daemon expone gRPC y el cliente delgado —sea laptop, móvil o navegador— consume ese mismo contrato.

*   **Posición sobre Web:** **diseño preparado, implementación futura incierta.** Si se materializa, será Flutter Web como cáscara delgada contra un Core Rust remoto por gRPC, idéntico contrato al móvil. No hay compromiso de fecha ni de alcance; se documenta para que el contrato gRPC no asuma un único tipo de cliente.

*   **Detección de muerte del proceso padre — mecanismo por plataforma (resuelve el gap de `prctl`):** un worker hijo debe morir solo si su proceso padre desaparece. El mecanismo canónico para **todas** las plataformas es el **archivo keepalive** que el worker sondea: cuando el archivo deja de existir, el worker termina. Latencia de detección ≈ el intervalo de sondeo (~50 ms), aceptable para la Fundación (EPIC-0).
    - **Linux:** además del keepalive, se permite la llamada al kernel `prctl(PR_SET_PDEATHSIG)` como **optimización opcional** que acelera la detección a inmediata. Es exclusiva de Linux.
    - **macOS:** **solo keepalive.** PROHIBIDO usar `prctl` (no existe en macOS). El bloque que la invoque debe estar acotado a `target_os = "linux"`, **NUNCA** a "cualquier Unix" — `#[cfg(unix)]` incluye macOS y por eso rompía la compilación.
    - **Windows:** **solo keepalive** (sin señales POSIX).
    - **Mejora diferida:** en macOS se puede añadir `kqueue(2)` + `EVFILT_PROC` para detección inmediata como mejora opcional en una fase madura (EPIC-8+). No se construye ahora.

*   **¿Qué restricciones tiene (lo que NUNCA puede pasar)?**
    - El Core Rust NUNCA corre en móvil ni en navegador.
    - El cliente delgado (móvil/Web) NUNCA aloja lógica de negocio; solo consume el contrato gRPC.
    - Una llamada al kernel exclusiva de un SO NUNCA vive bajo un `cfg` más amplio que ese SO. macOS y Windows deben compilar siempre.
    - Ninguno de los tres escritorios (Windows/Linux/macOS) es "secundario": el Core debe compilar y pasar pruebas en los tres.

*   **¿Cómo se vería en el sistema (efecto observable)?**
    - El binario Core compila y pasa la batería de pruebas en Windows, Linux y macOS sin `cfg` que excluya a Mac de rutas comunes.
    - El usuario abre la app móvil, apunta a su VPS o a su escritorio por red local, y opera contra el mismo Core sin que el teléfono ejecute cómputo pesado.
    - Si el proceso padre de un worker muere, el worker termina por sí mismo en cualquier SO (vía keepalive), sin dejar procesos huérfanos.

*   **¿Qué cuesta (trade-off)?** El keepalive paga ~50 ms de latencia de detección frente a la detección inmediata de `prctl`; se acepta porque es portátil y ya está probado. Mantener paridad en tres escritorios obliga a una matriz de compilación más amplia (más superficie de CI), a cambio de no tener una plataforma rota por sorpresa.

*   **Trazabilidad:** ADR-0033 (modos de despliegue que esta matriz concreta a nivel de SO), ADR-0016 y ADR-0030 (Local-First y Zero-Docker, agnósticos al SO, que esta decisión hace explícitos sobre los tres escritorios), ADR-0119 (topología distribuida que el cliente móvil reutiliza), ADR-0029 (acoplamiento FFI Rust+Flutter), ADR-0123 (canales de acceso al Core). Origen: Orden de trabajo [TASK-012](../execution/TASK-012-cross-platform-deployment-adr.md).
